# Audit technique — Juin 2026

Audit complet de la plateforme (Symfony 6.4 / PHP 8.3) : pérennité, qualité du code, architecture, sécurité, tests, CI/CD, évolutivité.

Périmètre mesuré : `src/` (28 770 lignes, 279 fichiers PHP), `legacy/` (6 854 lignes, 69 fichiers), 137 templates Twig, 83 fichiers de test, 154 migrations.

> **Fil conducteur — les garde-fous.** Le niveau de compétence des contributeurs (bénévoles) est très hétérogène. La bonne réponse n'est pas « faire plus attention » mais **rendre les erreurs impossibles ou bloquées par la machine**. Les recommandations sont donc priorisées autour de garde-fous automatisés plutôt que de bonnes pratiques reposant sur la vigilance humaine.

## Vue d'ensemble

| Dimension | Verdict |
|---|---|
| Stack & socle moderne | ✅ Sain. Symfony 6.4 LTS, PHP 8.3, API Platform, Doctrine, JWT RS256, 26 voters, autoescape Twig, front Vue/Vite/TS à jour |
| Frontière legacy | 🔴 Risque dominant. 6,8k lignes encore servies (46 routes), double couche DB, injections SQL résiduelles |
| Sécurité | 🟠 Code moderne propre ; risques concentrés dans le legacy + 1 webhook |
| Tests & qualité statique | 🟠 Tests bons là où ils existent, mais trous sur parcours financiers ; PHPStan niveau 1 (trop bas) |
| CI/CD & déploiement | 🔴 Gates non garantis bloquants, token unique partagé, pas de rollback |
| Pérennité humaine | 🟠 Bus factor ~1, activité en déclin depuis mars 2026 |
| Évolutivité multi-clubs | ✅ Architecture par env saine, quasi sans couplage dur |

**Diagnostic en une phrase** : le cœur est moderne et bien conçu, mais le projet est exposé sur trois fronts — le legacy encore branché en prod, une chaîne de livraison sans garde-fous bloquants, et une dépendance humaine à 1-2 personnes. Ces risques se renforcent : peu de mainteneurs + contributeurs hétérogènes + filet de sécurité lâche = chaque PR peut introduire une régression non détectée.

---

## 🔴 Priorité 1 — Garde-fous de livraison

Aujourd'hui, rien ne garantit qu'un contributeur peu expérimenté ne casse pas la prod.

1. **Branch protection bloquante** sur `main` et `production` : rendre `PHPUnit`, `PHPStan`, `PHP CS Fixer`, `Playwright` *required status checks*. Sans cela, la CI est purement consultative — un merge avec CI rouge reste possible. **À vérifier d'urgence dans les settings GitHub.**
2. **Le déploiement ne rejoue aucun test** (`.github/workflows/deploy-clevercloud.yml`) et se déclenche sur push `production` avec `force: true`. Un hotfix poussé directement part en prod sans vérification. → Ajouter un gate, ou interdire le push direct sur `production`.
3. **Pas de rollback** : un échec = notification Slack, revert manuel. → Au minimum un bouton `workflow_dispatch` « redéployer le SHA précédent » (les variables `*_LAST_DEPLOYED_SHA` existent déjà).
4. **Token Clever Cloud unique, nominatif, annuel, partagé tous clubs** : SPOF organisationnel. Expiration = tous les déploiements bloqués ; compromission = toutes les apps. → Token de service par club + alerte d'expiration.
5. **CODEOWNERS + revue obligatoire** sur les zones sensibles : `legacy/`, `config/packages/security.yaml`, paiements, `migrations/`.

## 🔴 Priorité 2 — La frontière legacy

Le legacy n'est pas en extinction passive : 173 commits sur 12 mois (mais seulement 2 sur les 3 derniers — l'extinction est enfin amorcée). Reste :

- **46 fichiers legacy exposés comme routes HTTP** (`src/Legacy/LegacyRouteLoader.php`), admin/CMS de contenu et commissions.
- **Double couche d'accès base** : Doctrine + `mysqli` brut (`src/Utils/MysqliHandler.php`, 41 fichiers legacy) sur les mêmes tables → risque d'incohérence de cache Doctrine.
- **Injections SQL résiduelles** (voir Priorité 3).
- **Couplage via service-locator global** : `LegacyContainer::get()` lit `$GLOBALS['kernel']`, 14 alias `legacy_*` publics dans `config/services.yaml`.

**Garde-fou** : check CI qui **interdit toute nouvelle ligne ajoutée dans `legacy/`** (`git diff` sur le périmètre → échec si insertions). On gèle, on ne migre plus que vers le bas. Le reliquat (CMS de contenu) est borné : chantier fini de quelques mois, les gros morceaux (droits, articles, users) étant déjà migrés.

> **Plan détaillé** : voir [Plan de démantèlement du legacy](plan-demantelement-legacy.md) — analyse bloc par bloc, ordre de suppression, estimation révisée (~30–43 j-h) et gains immédiats (~133 LOC supprimables sans migration).

## 🔴 Priorité 3 — Sécurité

Le code moderne est sain : mots de passe `#[Ignore]` en API, 26 voters, CSRF actif sur `form_login`, JWT RS256, contrôle objet `object == user`. Les vrais points sont dans le legacy + un webhook.

| Sévérité | Faille | Emplacement |
|---|---|---|
| Élevé | SQLi par nom de table/colonne (`escapeString` n'échappe pas les identifiants/backticks) | `legacy/scripts/operations.php:76-92` |
| Élevé | Path traversal à l'upload (`$_GET['dossier']` non assaini) | `legacy/valums-file-upload/server/admin-upload.php:24` |
| Élevé | Webhook HelloAsso : signature **contournable** en omettant le header → fraude au paiement | `src/Controller/HelloAssoWebhookController.php:39-59` |
| Élevé | CSRF sur suppression de fichiers via GET | `legacy/admin/ftp-deletefile.php` |
| Moyen | XSS réfléchi + SQLi authentifiés (`ROLE_CONTENT_MANAGER`) | `legacy/editElt.php:92,197` |
| Moyen | `.env` versionné (placeholders aujourd'hui — à confirmer surchargés en prod) | `.env` |
| Moyen | Upload validé par extension uniquement (pas de MIME) | `legacy/valums-file-upload/server/vfu.classes.php:117` |
| Faible | `composer audit` : 47 advisories (1 critique, 9 high) | `composer.lock` |
| Faible | `switch_user` activé sans rôle explicite restrictif | `config/packages/security.yaml:57` |

**Garde-fous** : `gitleaks` en pre-commit + CI (repo public !), `composer audit` en gate CI, headers de sécurité (CSP/HSTS via `nelmio/security-bundle`), signature HelloAsso rendue obligatoire, `samesite=lax` sur les cookies.

> **Faux positifs vérifiés (ne pas remédier)** : scripts `legacy/scripts/*` non routables (exclus par `LegacyRouteLoader`, gardés par le dispatcher `operations.php` derrière `ROLE_CONTENT_MANAGER`) ; `comment.html.twig` et `right-column.html.twig` sont autoescapés par Twig (pas de `|raw`). L'effort de pontage sécurité Symfony→legacy a été fait sérieusement.

## 🟠 Priorité 4 — Filet qualité automatisé

- **PHPStan niveau 1** alors que `phpstan.neon` déclare `level: 0` (incohérence ; la CI exécute `-l 1`). Beaucoup trop bas pour 28k lignes gérant paiements et données personnelles. → **Monter vers niveau 5+ avec `phpstan-baseline.neon`** pour figer la dette legacy sans bloquer. C'est le garde-fou qui rattrape le plus d'erreurs de débutant (types, null, méthodes inexistantes).
- **Couverture à trous sur l'argent et l'accès** : reset de mot de passe non testé, 10/11 validateurs de notes de frais non testés (`src/Validator/ExpenseReport/`), aucun E2E sur l'inscription à une sortie (cœur de la plateforme).
- **Migrations** : 21/154 ont un `down()` vide, 17 contiennent des `DROP/RENAME`. → Check CI interdisant une nouvelle migration destructive sans `down()`.
- **Qualité réelle des tests existants** : bonne (mock expectations sérieuses, chemins d'erreur webhook couverts, 0 test désactivé, lutte active documentée contre la flakiness E2E). Paiements et sync FFCAM bien testés. À préserver.
- Aligner la version MySQL dev/CI (`8.0.33` partout, y compris `prepare-for-test/action.yml`).

## 🟠 Priorité 5 — Architecture & lisibilité

Réduire la barrière à l'entrée pour les nouveaux contributeurs est en soi un garde-fou.

- **Contrôleurs obèses** : `SortieController` 1396 l. (`join()` ~256 l. de métier inline, `create()` ~290 l.), `UserController` 1022 l. → Extraire `EventRegistrationService` / `EventLifecycleService`. Bénéfice : testable **et** plus accessible.
- **Collision de vocabulaire sur le concept central** : entité `Evt`, form `EventType`, contrôleur `SortieController`, API `shortName: sortie`, variables `$evt`/`$event`/`$sortie` mélangées + propriétés FR/EN. Charge cognitive permanente. → Décider d'un nom canonique (recommandation : `Event` en code, « Sortie » en libellé UI/URL) + convention de langue documentée.
- **Constructeur 13-args de `Evt`** (`src/Entity/Evt.php:279`) + champs morts (`groupe`, `massif`). → Constructeur minimal + setters/DTO.
- **Duplication** : ~14 blocs de contexte email dans `SortieController` → `EventMailContextBuilder`.
- **TODO d'autorisation non traité** : `src/State/ExpenseReportCreateProcessor.php:42` (création de note de frais sans vérif du groupe encadrant) — quick win.
- **Codes de droit en chaînes magiques** : ~60 `UserRights::allowed('...')` non centralisés → enum + voters.
- Points sains : API Platform exemplaire sur `ExpenseReport` (DTO+Processor), entités `Article`/`Page` propres, namespaces standard.

## 🟠 Priorité 6 — Pérennité dépendances & humaine

- **Symfony 6.4 → 7.x à planifier avant nov. 2027** (fin support sécu). Doctrine ORM 2→3 et API Platform 3→4 en retard (à séquencer : AP4 demande souvent ORM3). `composer/package-versions-deprecated` **abandonnée** → à retirer (quick win). SDK Sentry/Firebase/JWT en retard de version majeure.
- **Front sain et à jour** : Vue 3.5, Vite 5.4, Tailwind 3.4, TS 5.5, Playwright 1.57.
- **Bus factor ~1** : sur 12 mois, 1 contributeur = 73 % des commits, et il a quasiment cessé depuis mars 2026 ; charge bascule sur 2 personnes ; rythme en déclin (pics à 90-104 commits/mois mi-2025 → 5-14 au printemps 2026). → Co-maintenance ≥2 personnes avec accès infra, transfert du « bus knowledge » (token, DB, déploiement), parcours « good first issue ».
- **Licence contradictoire** : badge MIT (README) vs `proprietary` (composer.json) vs `UNLICENSED` (package.json) sur un **repo public** → ambiguïté juridique réelle. Trancher et harmoniser.
- **README** annonce PHP 8.2, le code exige 8.3.
- **Multi-clubs sain** : multi-tenant par variables d'env, couplage dur résiduel limité à `src/Form/EventType.php` (valeurs `chambery`/`clermont` en dur).

## Annexe — Garde-fous CI prêts à implémenter

### A. Empêcher tout nouveau code dans `legacy/`

Job GitHub Actions (sur `pull_request`) :

```yaml
no-new-legacy-code:
  runs-on: ubuntu-latest
  steps:
    - uses: actions/checkout@v4
      with:
        fetch-depth: 0
    - name: Interdire les ajouts dans legacy/
      run: |
        ADDED=$(git diff --numstat origin/${{ github.base_ref }}...HEAD -- 'legacy/**' \
          | awk '{ added += $1 } END { print added+0 }')
        if [ "$ADDED" -gt 0 ]; then
          echo "::error::$ADDED ligne(s) ajoutée(s) dans legacy/. Le legacy est gelé : migrez vers src/."
          exit 1
        fi
```

### B. Interdire une migration destructive sans `down()`

```yaml
migrations-reversibles:
  runs-on: ubuntu-latest
  steps:
    - uses: actions/checkout@v4
      with:
        fetch-depth: 0
    - name: Vérifier down() sur les nouvelles migrations destructives
      run: |
        FILES=$(git diff --name-only --diff-filter=A origin/${{ github.base_ref }}...HEAD -- 'migrations/*.php')
        for f in $FILES; do
          if grep -qiE 'DROP (TABLE|COLUMN)|RENAME (TABLE|COLUMN|INDEX)' "$f"; then
            # down() doit contenir au moins un addSql/execute
            if ! awk '/function down/,/^    }/' "$f" | grep -qE 'addSql|->execute'; then
              echo "::error file=$f::Migration destructive sans down() réversible."
              exit 1
            fi
          fi
        done
```

### C. Scan de secrets (gitleaks)
